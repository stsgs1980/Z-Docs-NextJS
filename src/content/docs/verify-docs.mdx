# verify-docs

**Data-driven doc consistency checker.** Verifies that numbers in your README actually match the codebase. Never ship stale documentation again.

```
=== 1. README vs Code ===

[OK] Routes: code=12 readme=12 MATCH
[OK] Components: code=47 readme=47 MATCH
[OK] Models: code=8 readme=8 MATCH
[ERR] Tests: code=34 readme=28 MISMATCH -> fix: 28 -> 34
[OK] Commits: code=156 readme=154 MATCH (±5)

!! 1 mismatch(es) found!
```

## The Problem

You write `"12 components"` in your README. Two weeks later there are 15. Six months later your README is a pack of lies. Every project has this problem. Nobody updates docs manually after every change.

## The Solution

`verify-docs` reads your source code, counts real numbers, and compares them with what your README claims. If something drifted — you know immediately.

- **Section 1:** README vs Code — catches every stale number
- **Section 2:** Cross-repo Consistency — catches drift between sibling repos (wiki, donors, mirrors)
- **Plugin system:** Register your own `custom:*` source types for project-specific logic

## Quick Start

### Option A: One-command setup (recommended)

```bash
# 1. Install globally
git clone https://github.com/stsgs1980/verify.git
cd verify && bun link

# 2. In any project — initialize
cd /path/to/your-project
verify-docs-init
```

`verify-docs-init` automatically:
- Creates `verify-docs.json` with default checks
- Installs `.git/hooks/pre-push` that blocks push if README is stale

After that, every `git push` is verified automatically.

### Option B: Manual setup

#### 1. Install

```bash
# As a workspace package (recommended for monorepos)
cp -r verify-docs/ packages/verify-docs/

# Or standalone
git clone https://github.com/stsgs1980/verify.git
cd verify && bun install
```

#### 2. Create config

Create `verify-docs.json` in your project root:

```json
{
  "readme": "README.md",
  "checks": [
    {
      "name": "Routes",
      "source": "glob:src/app/**/page.tsx",
      "readmePattern": "(\\d+) routes"
    },
    {
      "name": "Models",
      "source": "file:prisma/schema.prisma",
      "countPattern": "^model \\w+",
      "readmePattern": "(\\d+) models"
    },
    {
      "name": "Commits",
      "source": "git:HEAD",
      "readmePattern": "(\\d+) commits",
      "tolerance": 5
    }
  ]
}
```

#### 3. Run

```bash
verify-docs                              # run with defaults
verify-docs --ci                         # CI mode (skip cross-repo)
verify-docs --config=my-config.json      # custom config path
```

#### 4. Automate (recommended)

Add a git pre-push hook so verification runs automatically before every push:

```bash
# Using the init command (easiest)
verify-docs-init

# Or manually
cp verify-docs/templates/pre-push .git/hooks/pre-push
chmod +x .git/hooks/pre-push
```

Or add to `package.json` for automatic installation:

```json
{
  "scripts": {
    "postinstall": "bun run verify-docs/templates/install-hooks.ts"
  }
}
```

Now every `git push` is blocked if your README lies. Bypass with: `git push --no-verify`

## Configuration Reference

### Root config (`verify-docs.json`)

| Field | Type | Required | Description |
|-------|------|----------|-------------|
| `readme` | string | yes | Path to the document to verify (relative to project root) |
| `checks` | CheckConfig[] | yes | Section 1 checks: code vs README |
| `crossRepo` | CrossRepoConfig[] | no | Section 2 checks: cross-repo consistency |

### CheckConfig

Each check counts something in your code and compares it with a number extracted from your README.

| Field | Type | Required | Description |
|-------|------|----------|-------------|
| `name` | string | yes | Human-readable name (displayed in output) |
| `source` | string | yes | Where to get the actual value. Built-in: `file:...`, `glob:...`, `git:HEAD`. Custom: `custom:...` |
| `countPattern` | string | no* | Regex pattern — count all matches in the source file (*required for `file:` source*) |
| `countExclude` | string[] | no | Exclude matches containing any of these strings |
| `exclude` | string[] | no | Exclude file paths containing any of these strings (*only for `glob:` source*) |
| `readmePattern` | string\|null | yes | Regex with `(\d+)` capturing group to extract number from README. `null` = info-only |
| `tolerance` | number | no | Allowed difference. Use for volatile numbers like commit count |
| `infoOnly` | boolean | no | If true, only print the value — don't compare with README |

### CrossRepoConfig

Cross-repo checks verify that numbers in sibling repos (wikis, donors, mirrors) match your project.

| Field | Type | Required | Description |
|-------|------|----------|-------------|
| `name` | string | yes | Human-readable name |
| `repo` | string | yes | Relative path to sibling repo from project root (e.g. `"../other-repo"`) |
| `source` | string | yes | File path within sibling repo. Prefix `"file:"` for clarity |
| `filePattern` | string | yes | How to extract a number: `"extract:PATTERN"` (first match group) or `"PATTERN"` (count all matches) |
| `matchAgainst` | string\|null | no* | Name of a Section 1 check to compare against |
| `readmePattern` | string\|null | no* | Alternative: extract expected value from README regex |
| `tolerance` | number | no | Allowed difference |

*Either `matchAgainst` or `readmePattern` must be provided.*

## Built-in Source Types

### `file:path` — Count regex matches in a file

Reads a file, applies `countPattern`, counts all matches.

```json
{
  "name": "Prisma models",
  "source": "file:prisma/schema.prisma",
  "countPattern": "^model \\w+",
  "readmePattern": "(\\d+) models"
}
```

With `countExclude` — filter out unwanted matches:

```json
{
  "name": "Scoring criteria",
  "source": "file:src/scoring/index.ts",
  "countPattern": "^\\s+(\\w+): number;",
  "countExclude": ["overall"],
  "readmePattern": "(\\d+)-criteria scoring"
}
```

### `glob:path` — Count files matching a pattern

Recursively finds files matching a simple glob pattern.

```json
{
  "name": "Tests",
  "source": "glob:src/**/*.test.ts",
  "readmePattern": "(\\d+) tests"
}
```

With `exclude` — skip certain paths:

```json
{
  "name": "Components",
  "source": "glob:src/components/**/*.tsx",
  "exclude": ["test", "spec", "__mocks__"],
  "readmePattern": "(\\d+) components"
}
```

### `git:HEAD` — Count commits

Returns the total commit count on the current branch.

```json
{
  "name": "Git commits",
  "source": "git:HEAD",
  "readmePattern": "\\((\\d+) commits\\)",
  "tolerance": 5
}
```

Always use `tolerance` for commit counts — they change on every push.

## Plugin System: Custom Source Types

When built-in sources aren't enough, create a `verify-docs.plugins.ts` in your project root:

```typescript
// verify-docs.plugins.ts
import { readFileSync, readdirSync, statSync } from "fs";
import { join } from "path";

export default function register({ registerSource }: {
  registerSource: (prefix: string, resolver: (source: string, root: string) => number | null) => void
}) {
  // Count workspace packages
  registerSource("custom:packages", (_source, root) => {
    try {
      return readdirSync(join(root, "packages"), { withFileTypes: true })
        .filter(d => d.isDirectory() && !d.name.startsWith("."))
        .filter(d => {
          try { return statSync(join(root, "packages", d.name, "package.json")).isFile(); }
          catch { return false; }
        })
        .length;
    } catch { return null; }
  });

  // Count Next.js pages (excluding dynamic routes)
  registerSource("custom:screens", (_source, root) => {
    const results: string[] = [];
    const dir = join(root, "src/app/(dashboard)");
    try {
      const entries = readdirSync(dir, { withFileTypes: true });
      for (const entry of entries) {
        if (entry.isDirectory() && !entry.name.startsWith("[")) {
          try {
            statSync(join(dir, entry.name, "page.tsx"));
            results.push(entry.name);
          } catch { /* no page.tsx */ }
        }
      }
    } catch { return null; }
    return results.length;
  });
}
```

Then use `custom:*` sources in your config:

```json
{
  "name": "Screens",
  "source": "custom:screens",
  "readmePattern": "Screens \\((\\d+)\\)"
}
```

The plugin file is automatically loaded from the project root when you run `verify-docs`.

## Cross-repo Checks

If your project has sibling repos (documentation wiki, donor/source repo, mirrors), Section 2 verifies consistency across them.

### How it works

1. Read a file from a sibling repo
2. Extract a number using `filePattern`
3. Compare against a value from Section 1 (`matchAgainst`) or from README (`readmePattern`)

### Example: Wiki consistency

```json
{
  "name": "Wiki formulas count",
  "repo": "../project-wiki",
  "source": "file:docs/api.md",
  "filePattern": "extract:Formulas \\((\\d+)\\)",
  "matchAgainst": "Formulas"
}
```

This reads `../project-wiki/docs/api.md`, extracts the number from `Formulas (10)`, and compares it with the "Formulas" check from Section 1.

### Example: Donor repo model count

```json
{
  "name": "Donor models",
  "repo": "../project-donor",
  "source": "file:prisma/schema.prisma",
  "filePattern": "^model \\w+",
  "readmePattern": "Prisma Schema \\((\\d+) models\\)"
}
```

This counts `model Xyz` lines in the donor's schema, then compares with the number in your README.

### Cross-repo chaining

Cross-repo results are added to the lookup. Later checks can reference earlier ones via `matchAgainst`:

```json
{
  "name": "Donor models",
  "repo": "../donor",
  "source": "file:prisma/schema.prisma",
  "filePattern": "^model \\w+",
  "matchAgainst": null,
  "readmePattern": "Prisma Schema \\((\\d+) models\\)"
},
{
  "name": "Wiki donor models",
  "repo": "../wiki",
  "source": "file:docs/schema.md",
  "filePattern": "extract:Prisma Schema \\((\\d+)\\)",
  "matchAgainst": "Donor models"
}
```

## CI Integration

### GitHub Actions

```yaml
# .github/workflows/verify.yml
name: Verify Docs

on:
  push:
    branches: [main, master]
  pull_request:
    branches: [main, master]

jobs:
  verify:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
        with:
          fetch-depth: 0  # full history for commit count

      - uses: oven-sh/setup-bun@v2

      - run: bun install --frozen-lockfile

      - name: Verify doc consistency
        run: bunx verify-docs --ci
```

The `--ci` flag skips cross-repo checks (sibling repos are not available in CI).

### Git pre-push hook

Automatically verify before every push. If docs are stale — push is blocked.

```bash
# 1. Create hook file
mkdir -p scripts/git-hooks
cat > scripts/git-hooks/pre-push << 'EOF'
#!/usr/bin/env bash
set -euo pipefail
cd "$(git rev-parse --show-toplevel)"
bunx verify-docs
if [ $? -ne 0 ]; then
  echo "PUSH BLOCKED: Fix numbers above or use --no-verify"
  exit 1
fi
EOF

# 2. Install it
cp scripts/git-hooks/pre-push .git/hooks/pre-push
chmod +x .git/hooks/pre-push
```

Or use the provided template:

```bash
cp verify-docs/templates/pre-push scripts/git-hooks/
bun run verify-docs/templates/install-hooks.ts
```

Bypass with: `git push --no-verify`

## Exit Codes

| Code | Meaning |
|------|---------|
| 0 | All numbers consistent |
| 1 | Mismatch found |
| 2 | Config error (missing file, bad JSON) |

## How readmePattern Works

The `readmePattern` is a regex that must contain a capturing group `(\d+)` (or `(\\d+)` in JSON). The first captured group is parsed as an integer.

| README text | readmePattern | Captured |
|-------------|---------------|----------|
| `Screens (19)` | `Screens \((\d+)\)` | 19 |
| `42 tests` | `(\d+) tests` | 42 |
| `Prisma Schema (36 models)` | `Prisma Schema \((\d+) models\)` | 36 |
| `6-criteria scoring` | `(\d+)-criteria scoring` | 6 |
| `v2.1 (156 commits)` | `\((\d+) commits\)` | 156 |

## Programmatic API

Use the engine directly in your own scripts:

```typescript
import { verify, registerSource } from "verify-docs/engine";
import { readFileSync } from "fs";

// Register custom sources
registerSource("custom:pages", (_source, root) => {
  // your logic here
  return 42;
});

// Load config and run
const config = JSON.parse(readFileSync("verify-docs.json", "utf-8"));
const result = verify(process.cwd(), config, { ci: false });

console.log(result.passed);   // boolean
console.log(result.errors);   // number of mismatches
console.log(result.section1); // LineResult[]
console.log(result.section2); // LineResult[]
```

## Use Cases

verify-docs is a standalone, project-agnostic tool. Here are typical scenarios:

| Scenario | What to check | Source type |
|----------|--------------|-------------|
| **Solo project** | Routes, components, tests count | `glob:`, `file:` |
| **Monorepo** | Workspace packages, shared modules | `custom:packages` plugin |
| **Multi-repo project** | Wiki vs code, donor vs target repo | `crossRepo` checks |
| **Open source** | Contributors, issues, PRs | `git:HEAD`, `custom:*` plugins |
| **CI/CD pipeline** | Every PR is verified automatically | `--ci` flag |

Any project with numbers in its README can use verify-docs. No framework or language restrictions.
