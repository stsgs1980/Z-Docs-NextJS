## 0. TL;DR -- Краткий справочник

**3 ключевых решения при выборе системы документации:**

1. **Парадигма** -- Wiki (сеть ссылок) или Docs (линейные документы) или Notes (атомарные заметки)
2. **Хранение** -- SaaS (облако) или Docs-as-Code (Git) или On-premise (свой сервер)
3. **Структура** -- Иерархия (дерево) или Сеть (граф) или Плоская (поиск + теги)

**5 вещей, которые нужно знать:**

1. Wiki и Docs -- это разные парадигмы: Wiki = связи между страницами, Docs = линейное чтение
2. GitHub Wiki -- для быстрых заметок, GitHub Pages -- для серьёзной документации
3. MkDocs -- генератор статических сайтов из Markdown, деплой на GitHub Pages
4. GitBook -- облачный WYSIWYG-редактор, платный, ниже порог входа
5. Divio -- 4 типа документации: Tutorial, How-to Guide, Reference, Explanation
6. Product-Embedded Docs -- документация внутри продукта (Next.js), максимум контроля, но дорого

## 1. Введение и цели

Этот стандарт описывает основные подходы к организации знаний и документации, классификацию инструментов, и критерии выбора платформы. Он обобщает практический опыт работы с различными системами документации и предоставляет структурированную основу для принятия решений.

**Цели:**

- Классифицировать архитектурные подходы к построению Wiki и баз знаний
- Описать парадигмы организации знаний (Wiki, Docs, Notes, Database)
- Сравнить инструменты документации (MkDocs, GitBook, Confluence и др.)
- Разграничить подходы GitHub Wiki и GitHub Pages
- Предоставить критерии выбора системы под конкретные задачи
- Обеспечить единый глоссарий и систему кросс-ссылок

**Аудитория:**

- Технические лидеры, принимающие решения о выборе платформы документации
- Разработчики, настраивающие Docs-as-Code пайплайн
- Технические писатели, мигрирующие между платформами
- Команды, проектирующие базу знаний с нуля

## 3. Архитектурные подходы к построению Wiki

### 3.1. Иерархическая (Древовидная)

Структура типа "папка - подпапка - страница", аналогичная файловой системе. Каждый документ находится ровно в одном месте иерархии, навигация осуществляется по вложенности.

**Характеристики:**

| Аспект | Описание |
|--------|----------|
| Структура | Дерево с корнем и ветвями |
| Гибкость | Низкая -- каждый документ в одном месте |
| Масштабируемость | Средняя -- при росте глубина увеличивается |
| Сложность поддержки | Низкая -- интуитивно понятна |

**Плюсы:**

- Интуитивно понятна для новых участников
- Легко навигировать по breadcrumbs
- Понятная модель прав доступа (на уровне разделов)
- Естественная для людей с файловой ментальной моделью

**Минусы:**

- Жёсткая структура -- сложно перекрёстно ссылаться
- Дублирование контента, когда тема относится к нескольким разделам
- Проблема "куда положить?" при неоднозначной классификации
- Глубокая вложенность делает навигацию утомительной

**Примеры:** Confluence (Spaces + Pages), SharePoint Wiki, Google Sites

**Типичная структура:**

```text
Проект/
  Документация/
    API/
      Эндпоинты/
      Аутентификация/
    Гайды/
      Быстрый старт/
      Конфигурация/
  Архитектура/
    Диаграммы/
    Решения/
```

### 3.2. Сетевая (Графовая / Link-based)

Страницы связываются двусторонними ссылками без жёсткой иерархии. Контент организован через связи, а не через вложенность. Каждая страница может ссылаться на любую другую.

**Характеристики:**

| Аспект | Описание |
|--------|----------|
| Структура | Направленный граф ссылок |
| Гибкость | Высокая -- любые связи между страницами |
| Масштабируемость | Высокая -- растёт органически |
| Сложность поддержки | Средняя -- нужна карта связей |

**Плюсы:**

- Максимальная гибкость -- контент не привязан к одному месту
- Естественно отражает взаимосвязи знаний
- Легко расширять без реорганизации
- Обратные ссылки (backlinks) показывают контекст

**Минусы:**

- Без визуальной карты графа легко потеряться
- Хаос при масштабировании без модерации
- Новичкам сложно понять структуру
- Нет единой "точки входа"

**Примеры:** Obsidian, TiddlyWiki, Roam Research, Logseq

### 3.3. Хаб-и-Споки (Hub-and-Spoke / MOC)

Есть центральные "хаб-страницы" (MOC -- Map of Content), от которых расходятся ссылки на детальные страницы. Каждый хаб -- тематический указатель, объединяющий все ссылки по одной теме.

**Характеристики:**

| Аспект | Описание |
|--------|----------|
| Структура | Гибрид: хабы (иерархия) + ссылки внутри (сеть) |
| Гибкость | Средняя -- иерархия хабов, свобода внутри |
| Масштабируемость | Высокая -- добавляются новые хабы |
| Сложность поддержки | Средняя -- нужно поддерживать хабы |

**Плюсы:**

- Баланс структуры и свободы
- Удобный вход в тему через хаб
- Обратные ссылки показывают контекст
- Легко добавлять новые страницы, не ломая навигацию

**Минусы:**

- Нужно вручную поддерживать хабы актуальными
- Хабы могут устаревать и вводить в заблуждение
- Нет автоматической категоризации

**Примеры:** Подход популярен в Obsidian/PKM-сообществе (MOC -- Map of Content)

**Пример хаба:**

```markdown
# API Documentation (MOC)

- [[Authentication]]
- [[Endpoints]]
- [[Rate Limiting]]
- [[Error Codes]]
- [[Webhooks]]
```

### 3.4. Плоская (Flat)

Все страницы на одном уровне, навигация осуществляется исключительно через поиск и теги. Никакой вложенности -- всё "в одной куче", а теги обеспечивают фильтрацию.

**Характеристики:**

| Аспект | Описание |
|--------|----------|
| Структура | Отсутствует -- все страницы равноправны |
| Гибкость | Максимальная -- нет проблемы "куда положить?" |
| Масштабируемость | Низкая -- без поиска полный хаос |
| Сложность поддержки | Низкая (при малом объёме) / Высокая (при большом) |

**Плюсы:**

- Максимум гибкости -- нет проблемы "куда положить?"
- Нет дублирования из-за множественной классификации
- Простая модель -- всё на одном уровне

**Минусы:**

- При росте объёма -- полный хаос без полнотекстового поиска
- Нет навигации по структуре
- Зависимость от качества тегирования
- Теги быстро превращаются в неуправляемый набор

**Примеры:** Ранние MediaWiki (Википедия -- почти плоская + категории), Notion без иерархии

### 3.5. Цеттелькастен (Zettelkasten)

Каждый "циеттель" (карточка) -- атомарная идея, одна мысль на страницу. Связи между карточками через идентификаторы и ссылки. Из атомарных заметок синтезируются более крупные работы.

**Характеристики:**

| Аспект | Описание |
|--------|----------|
| Структура | Граф атомарных карточек |
| Гибкость | Высокая -- произвольные связи |
| Масштабируемость | Средняя -- нужен строгий процесс |
| Сложность поддержки | Высокая -- дисциплина_required |

**Плюсы:**

- Глубина мысли -- каждая идея проработана
- Удобно для исследований и writing
- Из карточек собираются статьи, книги, доклады
- Связи порождают новые инсайты

**Минусы:**

- Высокий порог входа -- нужна дисциплина
- Не подходит для корпоративной документации
- Сложно найти "готовый ответ" -- нужно собирать из карточек
- Требует постоянного поддержания связей

**Примеры:** Obsidian, Roam Research, Logseq

### 3.6. Корпоративная (Топ-даун)

Структура задаётся сверху: по отделам, проектам, процессам. Жёсткие шаблоны страниц, ролевой доступ, регламенты редактирования. Иерархия определяется организационной структурой компании.

**Характеристики:**

| Аспект | Описание |
|--------|----------|
| Структура | Дерево по оргструктуре |
| Гибкость | Минимальная -- шаблоны и регламенты |
| Масштабируемость | Высокая -- растёт с компанией |
| Сложность поддержки | Высокая -- нужна модерация |

**Плюсы:**

- Порядок и контроль
- Соответствие корпоративным политикам
- Ролевой доступ по отделам
- Единые шаблоны обеспечивают консистентность

**Минусы:**

- Медленно эволюционирует
- "Могила" устаревших страниц без владельца
- Сложность внесения изменений для рядовых сотрудников
- Часто превращается в формальность

**Примеры:** Confluence с жёстким Space-структурированием, SharePoint, Notion Enterprise

### 3.7. Органическая (Боттом-ап)

Структура вырастает снизу -- каждый пишет, как считает нужным. Со временем стихийно формируются кластеры и привычки. Нет единого плана, структура возникает из практики.

**Характеристики:**

| Аспект | Описание |
|--------|----------|
| Структура | Хаос, переходящий в кластеры |
| Гибкость | Максимальная -- полная свобода |
| Масштабируемость | Низкая -- энтропия растёт |
| Сложность поддержки | Высокая -- нужна периодическая реорганизация |

**Плюсы:**

- Быстро стартует -- не нужно проектировать структуру
- Отражает реальные рабочие процессы
- Люди документируют то, что им действительно нужно

**Минусы:**

- Через полгода-год -- энтропия и "кладбище знаний"
- Дублирование и противоречия
- Новичкам невозможно разобраться
- Нет единого стиля и качества

**Примеры:** Так часто выглядит Wiki в стартапах без модерации, Notion без навигации

### 3.8. Docs-as-Code

Wiki хранится как Markdown-файлы в Git-репозитории. Изменения через Pull Request, ревью, версионирование. Документация -- часть кодовой базы, с тем же процессом и дисциплиной.

**Характеристики:**

| Аспект | Описание |
|--------|----------|
| Структура | Файловая система + Git |
| Гибкость | Средняя -- изменения через PR |
| Масштабируемость | Высокая -- ветки, теги, версии |
| Сложность поддержки | Средняя -- нужен CI/CD пайплайн |

**Плюсы:**

- Полноценный контроль версий (git blame, diff, revert)
- CI/CD для документации (автодеплой, линтинг, проверка ссылок)
- Интеграция с кодом -- документация рядом с исходниками
- PR-ревью обеспечивает качество
- Бесплатный хостинг (GitHub Pages, Netlify)

**Минусы:**

- Порог входа для нетехнических участников (нужен Git)
- Нет WYSIWYG-редактора
- Сложнее для продакт-менеджеров и техрайтеров без Git-навыков
- Нужно настраивать CI/CD

**Примеры:** MkDocs, Docusaurus, GitBook (с Git-синхронизацией), Hugo

## 4. Парадигмы организации знаний

### 4.1. Классификация по парадигме

Wiki и Docs -- это не одно и то же. Это разные парадигмы работы с информацией, и понимание различий критично для выбора платформы. Ниже приведена классификация, часто встречающаяся в PKM-сообществе (Personal Knowledge Management).

| Парадигма | Суть | Навигация | Роль автора |
|-----------|------|-----------|-------------|
| **Wiki** | Взаимосвязанные страницы через ссылки, открытое редактирование | По ссылкам и обратным ссылкам | Соавтор, редактор |
| **Docs** | Линейные структурированные документы (гайды, мануалы) | По оглавлению и иерархии | Писатель, рецензент |
| **Notes** | Личные быстрые заметки, атомарные мысли | Поиск, теги, обратные ссылки | Личный архив |
| **Database** | Структурированные записи, таблицы, фильтры | Фильтры, сортировка, виды | Куратор данных |
| **Handbook** | Нормативный справочник: политики, процедуры, онбординг | По регламенту и индексу | Редактор, утверждающий |

### 4.2. Разница между Wiki и Docs

Это ключевое различие, которое часто упускают при выборе платформы:

| Аспект | Wiki | Docs |
|--------|------|------|
| Модель чтения | Нелинейная -- прыжки по ссылкам | Линейная -- чтение сверху вниз |
| Структура | Сетевая -- страницы равноправны | Иерархическая -- главы и разделы |
| Авторство | Коллективное -- любой может редактировать | Регламентированное -- автор и ревьюер |
| Цель | Связать знания | Передать знания последовательно |
| Аудитория | Участники команды | Читатели (клиенты, новичков) |
| Примеры | Wikipedia, Confluence Wiki | ReadTheDocs, MkDocs-сайт |

### 4.3. Notion-стиль: платформа объединяет парадигмы

Современные платформы (Notion, Coda, Anytype) объединяют несколько парадигм в одном интерфейсе, позволяя использовать разные модели в разных пространствах:

| Парадигма | Реализация в Notion |
|-----------|---------------------|
| Doc | Страница с текстом, заголовками, блоками |
| Wiki | Пространство с обратными ссылками |
| Project | Доски, таск-трекинг, канбан |
| Database | Таблицы, канбан-вид, календарь, галерея |

## 5. Инструменты документации

### 5.1. MkDocs

**Что это:** Статический генератор сайтов для документации, написанный на Python. Превращает Markdown-файлы в красивый HTML-сайт с навигацией, поиском и темами.

**Как работает:**

1. Контент пишется в Markdown-файлах (.md)
2. Структура конфигурируется в одном файле mkdocs.yml
3. Команда `mkdocs build` генерирует готовый HTML-сайт
4. Команда `mkdocs serve` запускает локальный превью с автоперезагрузкой

**Структура проекта:**

```text
my-docs/
  mkdocs.yml          # конфигурация
  docs/
    index.md           # главная страница
    guide/
      install.md
      config.md
    api/
      endpoints.md
```

**Ключевые фишки:**

- Material for MkDocs -- самая популярная тема (поиск, навигация, тёмная тема, вкладки, адмонции)
- Деплой в одну команду: `mkdocs gh-deploy` -- сразу на GitHub Pages
- Версионирование документации через плагин mike
- Полноценный поиск, i18n, кастомные плагины

**Типичный пайплайн:**

```bash
# Установка
pip install mkdocs-material

# Локальная разработка
mkdocs serve

# Деплой на GitHub Pages
mkdocs gh-deploy
```

**Для кого:** Разработчики, которые хотят документацию рядом с кодом в Git-репозитории, с PR-ревью и CI/CD.

### 5.2. GitBook

**Что это:** Облачная платформа для создания и хостинга документации с визуальным WYSIWYG-редактором. Позволяет писать документацию в браузере без знания Markdown.

**Как работает:**

- Работа в браузере -- WYSIWYG-редактор (не нужно писать Markdown вручную)
- Можно подключить GitHub/GitLab -- контент синхронизируется с репозиторием
- Командная работа: ревью, комментарии, роли, пространства

**Ключевые фишки:**

- Красивый UI "из коробки"
- Пространства (Spaces) для разных продуктов/команд
- Интеграции: GitHub, GitLab, Slack, API
- API-документация из OpenAPI/Swagger
- Платный -- бесплатный тариф сильно урезан (1 пространство)

**Для кого:** Продакт-менеджеры, техрайтеры, команды без сильных Git-навыков.

### 5.3. Сравнение MkDocs и GitBook

| Аспект | MkDocs | GitBook |
|--------|--------|---------|
| Где работает | Локально / CI/CD | В браузере (SaaS) |
| Редактирование | Markdown + Git | WYSIWYG + Git (опционально) |
| Хостинг | Любой (GitHub Pages, Netlify, свой) | gitbook.io (или свой через Docker) |
| Цена | Бесплатно (open source) | Freemium (бесплатно -- 1 space) |
| Контроль | Полный (твои файлы) | Данные на платформе |
| Порог входа | Средний (нужен Python, Git) | Низкий (открыл и пишешь) |
| Для кого | Разработчики, DevOps | Продакт-менеджеры, техрайтеры |

### 5.4. Другие инструменты

| Инструмент | Тип | Особенность |
|------------|-----|-------------|
| **Docusaurus** | Docs-as-Code (React) | От Meta, MDX + React-компоненты |
| **ReadTheDocs** | Хостинг + CI | Автобилд из GitHub, бесплатный для open source |
| **Hugo** | Статический генератор | Очень быстрый, написан на Go |
| **Notion** | SaaS Wiki | Базы данных, гибкость, командная работа |
| **Confluence** | Корпоративная Wiki | Atlassian-экосистема, Jira-интеграция |
| **Jekyll** | Статический генератор | По умолчанию для GitHub Pages, Ruby |
| **Docsify** | Docs-as-Code (runtime) | Без генерации -- рендерит Markdown на лету |

## 6. GitHub Wiki vs GitHub Pages

### 6.1. GitHub Wiki

**Что это:** Встроенный в каждый репозиторий раздел с вики-страницами. Доступен на вкладке `Wiki` в интерфейсе GitHub.

**Характеристики:**

| Аспект | Детали |
|--------|--------|
| Где живёт | Вкладка Wiki в репозитории |
| Формат | Markdown, MediaWiki, Rdoc, Org-mode |
| Редактирование | Прямо в браузере или через Git (wiki.git) |
| Структура | Плоская + боковое меню (sidebar) + подвал (footer) |
| Сайдбар | `_Sidebar.md` -- кастомная навигация |
| Версионирование | Есть история правок (Git) |
| URL | `github.com/user/repo/wiki` |

**Плюсы:**

- Рядом с кодом, не нужно ничего настраивать
- Быстро начать -- нажал "Create Wiki" и пишешь
- История изменений встроена
- Поддерживает Markdown и другие форматы

**Минусы:**

- Минимальная кастомизация UI
- Нет полнотекстового поиска
- Нет тем, плагинов, красивой навигации
- Отдельный Git-репозиторий (wiki.git) -- не в основном коде
- Нет CI/CD, нет автодеплоя
- Привязан к GitHub -- нельзя легко перенести

### 6.2. GitHub Pages

**Что это:** Бесплатный хостинг статических сайтов прямо из репозитория GitHub. Поддерживает Jekyll по умолчанию, но можно использовать любой генератор.

**Характеристики:**

| Аспект | Детали |
|--------|--------|
| Где живёт | Ветка `gh-pages` или папка `/docs` в `main` |
| Формат | Любой -- Markdown - HTML через генератор |
| Редактирование | Через Git + генератор |
| Структура | Полная свобода -- любая архитектура |
| Кастомизация | Темы, плагины, свой домен, HTTPS |
| URL | `user.github.io/repo` или свой домен |
| CI/CD | GitHub Actions -- автобилд и деплой |

**Плюсы:**

- Красивый, профессиональный сайт документации
- Полный контроль над дизайном и структурой
- Поиск, навигация, версии, i18n -- через генератор
- Свой домен (`docs.myproject.com`)
- Бесплатный хостинг
- Работает с MkDocs, Jekyll, Docusaurus и т.д.

**Минусы:**

- Нужен генератор (MkDocs/Jekyll) -- порог входа выше
- Билд-процесс (CI/CD или ручной `mkdocs gh-deploy`)
- Контент отдельно от кода (если папка /docs, то рядом)

### 6.3. Сравнение

| Аспект | GitHub Wiki | GitHub Pages |
|--------|-------------|--------------|
| Внешний вид | Базовый GitHub-стиль | Красивый, кастомный |
| Поиск | Нет | Есть (через генератор) |
| Навигация | Sidebar + ссылки | Полная иерархия, хлебные крошки |
| Настройка | Минимальная | Полная |
| Сложность | Просто | Средне (нужен генератор) |
| Рядом с кодом | Отдельный git-репо | Ветка или /docs |
| Свой домен | Нет | Да |
| Версионирование | История правок | mike / ветки / теги |
| Для кого | Быстрые заметки, мелкие проекты | Серьёзная документация |

### 6.4. Когда что выбирать

| Ситуация | Выбор |
|----------|-------|
| Мелкий проект, пара страниц | GitHub Wiki |
| Нужна красивая документация | GitHub Pages + MkDocs |
| Open source с API-доками | GitHub Pages + Docusaurus |
| Быстро набросать заметки для команды | GitHub Wiki |
| Продакшн-документация продукта | GitHub Pages |
| Нужен поиск, i18n, версии | GitHub Pages (Wiki этого не умеет) |

## 7. Классификация Divio: 4 типа документации

### 7.1. Обзор системы

Divio (основана Daniele Procida) предложила одну из самых влиятельных классификаций документации. Она разделяет документацию на 4 типа по двум осям: ориентация (практическая / теоретическая) и этап работы (обучение / работа).

```text
                   Практическая          Теоретическая
               +-------------------+-------------------+
  Обучение     |    Tutorial       |   Explanation     |
               +-------------------+-------------------+
  Работа       |  How-to Guide     |   Reference       |
               +-------------------+-------------------+
```

### 7.2. Tutorial (Обучение)

| Аспект | Описание |
|--------|----------|
| Ориентация | Практическая, обучение |
| Цель | Довести новичка от нуля до первого результата |
| Форма | Пошаговое руководство, линейное |
| Тон | Как учебник -- ведёт за руку |
| Пример | "Быстрый старт: разворачиваем проект за 5 минут" |

**Ключевые принципы:**

- Линейный путь от начала до конца
- Один конкретный результат в конце
- Не объясняет "почему" -- только "делай так"
- Минимум отвлечений и альтернатив
- Проверяется на реальных новичках

### 7.3. How-to Guide (Руководство)

| Аспект | Описание |
|--------|----------|
| Ориентация | Практическая, решение задачи |
| Цель | Помочь решить конкретную практическую проблему |
| Форма | Рецепт, шаги для конкретного случая |
| Тон | Как кулинарная книга -- выбрал рецепт, приготовил |
| Пример | "Как деплоить в Kubernetes" |

**Ключевые принципы:**

- Ориентирован на задачу, а не на инструмент
- Предполагает базовое понимание продукта
- Можно входить в любой точке (не линейный)
- Фокус на конкретном случае, не на общем принципе

### 7.4. Reference (Справочник)

| Аспект | Описание |
|--------|----------|
| Ориентация | Теоретическая, факты |
| Цель | Предоставить точную информацию о механизмах |
| Форма | Сухое перечисление, API-спецификация |
| Тон | Как словарь -- точный, бесстрастный |
| Пример | Список REST-эндпоинтов, конфиг-параметры |

**Ключевые принципы:**

- Структурирован вокруг кода/API, не вокруг задач
- Однообразный формат для всех записей
- Не объясняет концепции -- только факты
- Автогенерируется из кода там, где возможно

### 7.5. Explanation (Объяснение)

| Аспект | Описание |
|--------|----------|
| Ориентация | Теоретическая, понимание |
| Цель | Помочь понять "почему так", контекст и мотивацию |
| Форма | Эссе, архитектурное решение, ADR |
| Тон | Как статья -- размышляющий, аргументированный |
| Пример | "Почему мы выбрали микросервисы вместо монолита" |

**Ключевые принципы:**

- Не привязан к конкретной задаче
- Расширяет понимание, а не решает проблему
- Показывает альтернативы и компромиссы
- Может быть нелинейным

## 9. Критерии выбора системы

### 9.1. Матрица решений

| Критерий | Wiki (Confluence) | Docs-as-Code (MkDocs) | SaaS (GitBook) | Notes (Obsidian) |
|----------|-------------------|----------------------|----------------|-------------------|
| Порог входа | Низкий | Средний | Низкий | Низкий |
| Качество UI | Среднее | Высокое (Material) | Высокое | Зависит от темы |
| Git-интеграция | Плагины | Нативная | Опциональная | Плагины |
| Поиск | Встроенный | Встроенный (Material) | Встроенный | Встроенный |
| Версионирование | История правок | Полный (Git) | История правок | Полный (Git) |
| Совместная работа | Нативная | Через PR | Нативная | Через Git |
| Стоимость | Платный | Бесплатно | Freemium | Бесплатно |
| Кастомизация | Средняя | Высокая | Низкая | Высокая |
| Офлайн-доступ | Нет | Нет | Нет | Да |

### 9.2. Алгоритм выбора

```text
1. Кто будет писать?
   - Только разработчики     -> Docs-as-Code (MkDocs, Docusaurus)
   - Смешанная команда        -> SaaS (GitBook, Notion)
   - Вся компания             -> Wiki (Confluence)

2. Какой объём?
   - Мелкий проект (до 20 страниц)   -> GitHub Wiki или README
   - Средний проект (20-200 страниц) -> MkDocs + GitHub Pages
   - Крупный продукт (200+ страниц)  -> Confluence или GitBook

3. Нужен ли свой домен и брендинг?
   - Да  -> MkDocs + GitHub Pages + свой домен
   - Нет -> GitHub Wiki или Confluence

4. Нужна ли интеграция с кодом?
   - Да  -> Docs-as-Code (MkDocs, Docusaurus)
   - Нет -> SaaS (GitBook, Notion)

5. Бюджет?
   - Нет бюджета    -> MkDocs + GitHub Pages (бесплатно)
   - Есть бюджет    -> Confluence, GitBook, Notion
```

### 9.3. Типичные паттерны

| Паттерн | Стек | Для кого |
|---------|------|----------|
| Open Source проект | MkDocs + Material + GitHub Pages + mike | Разработчики |
| Стартап | Notion (Wiki + Database) | Вся команда |
| Enterprise | Confluence + Jira | Крупные компании |
| API-платформа | Docusaurus + Swagger + ReadTheDocs | API-потребители |
| Личная база знаний | Obsidian + Git | Один пользователь |
| Inner Source | MkDocs + GitHub Enterprise | Внутренние разработчики |

## 10. Практические рекомендации

### 10.1. Миграция между платформами

Миграция документации -- сложный процесс, который часто недооценивают. Ключевые шаги:

1. **Аудит текущего состояния** -- посчитать объём, выявить устаревшее, оценить качество
2. **Выбор целевой структуры** -- определить парадигму (раздел 3) и инструмент (раздел 5)
3. **Экспорт контента** -- выгрузить в промежуточный формат (обычно Markdown)
4. **Трансформация** -- конвертировать форматы, починить ссылки, обновить навигацию
5. **Импорт и настройка** -- загрузить в новую систему, настроить CI/CD
6. **Перенаправление** -- настроить редиректы со старых URL

### 10.2. Гибридные подходы

На практике чаще всего встречаются гибридные варианты:

- **Иерархия верхнего уровня + сетевые ссылки внутри** -- самый частый паттерн
- **Хабы внутри древовидной структуры** -- MOC + иерархия
- **Docs-as-Code для API + Wiki для процессов** -- два инструмента рядом
- **Confluence для бизнес-документации + MkDocs для технической**

### 10.3. Антипаттерны

| Антипаттерн | Описание | Решение |
|-------------|----------|---------|
| Wiki без владельца | Никто не отвечает за раздел | Назначить владельца каждого раздела |
| Документация-кладбище | Устаревшие страницы без обновления | Регулярный аудит, дедлайны актуальности |
| Копипаст вместо ссылок | Дублирование контента | Использовать ссылки и transclusion |
| Все пишут как хотят | Нет шаблонов и стандартов | Ввести шаблоны и ESLint-проверку |
| Документация отдельно от кода | Доки устаревают при изменении кода | Docs-as-Code: доки рядом с кодом |

### 13.3. Case Study: Миграция документации Z-ai-platform с Confluence на MkDocs

#### 13.3.1. Контекст

Компания Z-ai разрабатывает платформу AI-агентов. На момент начала проекта документация была размазана по трём системам:

- Confluence -- бизнес-документация, онбординг, процессы (около 180 страниц)
- README.md в репозиториях -- техническая документация (разного качества, часто устаревшая)
- Google Docs -- черновики RFC и архитектурные решения (без версионирования)

**Проблемы:**

- Документация в Confluence устаревала: никто не обновлял страницы после изменения кода
- README-файлы не имели единого стиля и навигации
- RFC в Google Docs терялись, не было связи с тикетами и кодом
- Поиск работал плохо: один и тот же термин мог встречаться в трёх системах
- Новым сотрудникам требовалось 2-3 недели, чтобы найти нужную информацию

#### 13.3.2. Анализ и решение

**Шаг 1: Определение парадигмы (раздел 4)**

| Тип контента | Парадигма | Инструмент |
|--------------|-----------|------------|
| API-документация | Docs (Reference) | MkDocs + Swagger |
| Архитектурные решения | Docs (Explanation) | MkDocs (ADR) |
| Онбординг | Docs (Tutorial) | MkDocs |
| Runbook / Incidents | Wiki | Confluence (оставлен) |
| Бизнес-процессы | Handbook | Confluence (оставлен) |

Решение: гибридный подход. Техническая документация переезжает в MkDocs (Docs-as-Code), бизнес-процессы и runbook остаются в Confluence.

**Шаг 2: Выбор архитектуры (раздел 3)**

Выбрана иерархическая структура верхнего уровня с хаб-страницами (MOC) внутри разделов:

```text
docs/
  index.md                          # Главная (Tutorial)
  getting-started/
    index.md                        # Хаб: Быстрый старт
    install.md
    first-agent.md
    configuration.md
  architecture/
    index.md                        # Хаб: Архитектура
    overview.md
    adr/                            # Architecture Decision Records
      001-monorepo.md
      002-agent-lifecycle.md
  api/
    index.md                        # Хаб: API Reference
    authentication.md
    endpoints/
      agents.md
      tasks.md
  guides/
    index.md                        # Хаб: How-to Guides
    deploy-to-k8s.md
    custom-tool.md
    monitoring.md
  standards/
    index.md                        # Хаб: Стандарты
    markdown-formatting.md          # STD-DOC-002
    no-unicode-policy.md            # STD-DOC-003
```

**Шаг 3: Выбор инструмента (раздел 5)**

MkDocs + Material for MkDocs. Обоснование:

| Критерий | MkDocs | Альтернатива |
|----------|--------|-------------|
| Контроль версий | Нативный Git | Confluence -- только история правок |
| Деплой | GitHub Actions + Pages | Confluence -- ручной |
| Качество UI | Material theme | Confluence -- среднее |
| Цена | Бесплатно | Confluence -- $7/пользователь/мес |
| Интеграция с кодом | Доки в /docs/ рядом с кодом | Confluence -- отдельно |
| PR-ревью | Доки проходят ревью как код | Confluence -- нет процесса |

#### 13.3.3. Реализация

**mkdocs.yml:**

```yaml
site_name: Z-ai Platform Documentation
site_url: https://docs.z-ai.dev
site_description: Документация платформы AI-агентов

theme:
  name: material
```

#### 13.3.4. Процесс миграции

| Этап | Длительность | Действия |
|------|-------------|----------|
| Аудит | 1 неделя | Инвентаризация всех страниц, пометка устаревших |
| Структура | 3 дня | Проектирование навигации, создание скелета |
| Экспорт | 3 дня | Конвертация Confluence - Markdown (pandoc + ручная чистка) |
| Наполнение | 2 недели | Перенос контента, обновление устаревшего |
| Ревью | 1 неделя | PR-ревью каждой страницы командой |
| Деплой | 1 день | Настройка CI/CD и деплой на docs.z-ai.dev |
| Перенаправление | 2 дня | Редиректы со старых URL Confluence |

**Итого:** 4-5 недель от старта до продакшна.

#### 13.3.5. Результаты

| Метрика | До (Confluence) | После (MkDocs) |
|---------|-----------------|----------------|
| Время на обновление доков | 2-3 дня после изменения кода | В том же PR, что и код |
| Доля устаревших страниц | ~40% | < 5% |
| Время онбординга нового разработчика | 2-3 недели | 1 неделя |
| Поиск | Не находил 30% релевантного | Находит всё (Material search) |
| Стоимость | $7/пользователь/мес | $0 (GitHub Pages) |
| Процесс обновления | Ручной, без ревью | PR + ревью + автодеплой |

#### 13.3.6. Извлечённые уроки

1. **Не мигрируй всё сразу** -- начни с самого критичного (API, Getting Started), остальное переноси постепенно
2. **Конвертация Confluence - Markdown всегда ручная** -- pandoc даёт 70%, остальное чистишь руками
3. **Структура важнее контента** -- потрать время на проектирование навигации до переноса
4. **Шаблоны -- обязательно** -- создай шаблоны для ADR, Tutorial, How-to и Reference
5. **CI/CD для доков -- не опция, а необходимость** -- без `--strict` флага MkDocs не поймёт, что страница не найдена
6. **Оставь Confluence для бизнес-процессов** -- не всё нужно переносить в Docs-as-Code

### 13.4. Case Study: Product-Embedded Docs -- документация внутри продукта (ZCode)

#### 13.4.1. Контекст

В предыдущих кейсах мы рассматривали документацию как отдельный ресурс: статический сайт на MkDocs, SaaS-платформа вроде GitBook или вики-движок. Существует принципиально иной подход -- встроить документацию непосредственно в веб-приложение продукта. В этом случае раздел `/docs` является частью основного фреймворка, а не отдельным генератором или платформой.

Пример такого подхода -- документация ZCode (zcode.z.ai/en/docs/welcome). ZCode представляет собой Agentic Development Environment (ADE), среду разработки на базе AI-агентов, заточенную под Long Horizon Tasks -- длительные многошаговые задачи. Документация ZCode не вынесена на отдельный поддомен или статический генератор, а является органичной частью основного веб-приложения.

#### 13.4.2. Технический стек

Документация ZCode реализована на следующем стеке:

| Компонент | Технология | Примечание |
|-----------|-----------|----------|
| Фреймворк | Next.js (App Router) | React Server Components |
| Рендеринг | SSR + SSG | Страницы генерируются на сервере |
| Стилизация | Tailwind CSS + CSS-модули | Dark mode по умолчанию |
| i18n | URL-префиксы `/en/`, `/cn/` | Встроенная интернационализация Next.js |
| Поиск | Встроенный `Cmd+K` | Индексация на стороне клиента |
| Маршрутизация | Файловая система Next.js | `/app/[lang]/docs/[slug]/page.tsx` |
| Хостинг | Vercel или собственная инфраструктура | Единый деплой с основным приложением |

Ключевое отличие от MkDocs/Hugo -- документация рендерится как часть React-приложения. Это значит, что страницы документации могут использовать любой React-компонент: интерактивные виджеты, живые примеры кода, виджеты статуса API, авторизацию и персонализацию.

#### 13.4.3. Структура навигации

Навигация ZCode организована по принципу sidebar-меню, типичному для продуктовой документации:

```text
Get Started
  Welcome to the New ZCode
  Install
  API Key Setup
  Feedback & Support
Core Features
  ZCode Agent
  Goal Mode
  Remote Control
  Task & File Management
  Bot Channel
  Edit History
  Command
  Plugin
  MCP Servers
  Subagents
  Skill
  Usage Stats
  Integration
  Safety Confirmation
ADE Tools
Support
  Keyboard Shortcuts
  FAQ (Q&A)
```

Структура следует классификации Divio: Get Started -- Tutorial, Core Features -- Explanation + Reference, ADE Tools -- How-to Guide. Наличие страницы Install с download-ссылками и версиями указывает на тесную интеграцию документации с продуктовой логикой -- генератор статических сайтов не может рендерить динамические download-линки без дополнительной логики.

#### 13.4.4. Сравнение подходов к доставке документации

Product-Embedded Docs -- это третий крупный подход к доставке документации наряду со статическими генераторами и SaaS-платформами:

| Аспект | Статический генератор (MkDocs) | SaaS-платформа (GitBook) | Product-Embedded (Next.js) |
|--------|-------------------------------|-------------------------|--------------------------|
| Где живёт | Отдельный repo, GitHub Pages | Сторонний хостинг | Внутри основного приложения |
| Порог входа | Низкий (Markdown) | Очень низкий (WYSIWYG) | Высокий (React/Next.js) |
| Стоимость разработки | Низкая | Средняя (подписка) | Высокая (фронтенд-разработчик) |
| Кастомизация | Средняя (тема, плагины) | Низкая (ограничения SaaS) | Полная |
| Динамический контент | Нет (статический HTML) | Ограничен | Да (React-компоненты) |
| Аутентификация | Сложная (нужен слой) | Встроена (платформа) | Нативная (сессия продукта) |
| i18n | Через плагин (mkdocs-i18n) | Встроена | Нативная (Next.js routing) |
| Поиск | Плагин (Material search) | Встроен | Кастомный (любой) |
| Цикл публикации | CI/CD (отдельный) | Push в платформу | Деплой приложения (единый) |
| Внесение правок | Markdown + PR (разработчик) | WYSIWYG (писатель) | Код + PR (разработчик) |
| Аналитика | Сторонняя (GA, Plausible) | Ограничена | Полная (общая с продуктом) |

#### 13.4.5. Преимущества Product-Embedded Docs

1. **Единый кодбейс** -- документация и продукт живут в одном репозитории. Нет проблемы рассинхронизации между репо документации и репо продукта. Изменение API и обновление документации происходят в одном PR.

2. **Полный контроль над UX** -- дизайн документации полностью совпадает с дизайном продукта. Пользователь не чувствует "перескока" между продуктом и внешней документацией. Можно использовать любые React-компоненты: интерактивные демо, виджеты статуса, live-примеры.

3. **Нативная аутентификация** -- документация может быть приватной или персонализированной без дополнительного слоя. Пользователь, авторизованный в продукте, автоматически авторизован в документации. Можно показывать разный контент разным ролям.

4. **Динамический контент** -- страницы могут рендерить актуальную версию продукта, статус API, региональные настройки. В примере ZCode страница Install показывает текущую версию приложения (v3.1.2) и ссылки на скачивание, которые обновляются автоматически при каждом деплое.

5. **Единая аналитика** -- все данные о посещении документации интегрированы в общую аналитику продукта. Можно отслеживать путь пользователя от документации до действия в продукте.

#### 13.4.6. Недостатки Product-Embedded Docs

1. **Высокая стоимость разработки** -- для создания и поддержки документации нужен фронтенд-разработчик, а не технический писатель с Markdown. Каждая новая страница -- это React-компонент (или MDX-файл с кастомной обработкой). Это увеличивает цикл внесения правок.

2. **Единый цикл публикации** -- обновление документации привязано к деплою всего приложения. Нельзя быстро исправить опечатку, не прогоняя полный CI/CD-пайплайн. В MkDocs исправление вливается через PR и деплоится за 2-3 минуты.

3. **Сложность для нетехнических авторов** -- технический писатель без опыта работы с React/Next.js не может самостоятельно вносить правки. В MkDocs или GitBook достаточно знать Markdown.

4. **Отсутствие встроенного PR-ревью контента** -- в Docs-as-Code подходе каждое изменение документации проходит через PR с ревью. В Product-Embedded Docs документация -- это код, но ревью зачастую фокусируется на логике компонентов, а не на качестве текста.

5. **Привязка к фреймворку** -- миграция с Next.js на другой фреймворк означает переписывание всей документации. В случае MkDocs контент (Markdown) отделён от генератора и легко переносится.

#### 13.4.7. Критерии выбора: когда использовать Product-Embedded Docs

| Условие | Рекомендация |
|---------|-------------|
| Продукт -- SaaS с веб-интерфейсом, документация -- часть пользовательского опыта | Product-Embedded Docs |
| Нужна приватная документация с ролевой моделью | Product-Embedded Docs |
| Документация должна показывать live-данные (статус, версия, конфиг) | Product-Embedded Docs |
| Команда -- 1-3 разработчика, нужен быстрый старт | MkDocs + GitHub Pages |
| Нетехнические авторы -- основной источник контента | GitBook или Confluence |
| Open source-проект с API-документацией | Docusaurus или MkDocs |
| Нужна независимая от продукта документация | Статический генератор (MkDocs/Hugo) |
| Бюджет на фронтенд-разработку документации ограничен | MkDocs (80% результата при 20% усилий) |

#### 13.4.8. Гибридный подход: MDX в Next.js

Существует промежуточный вариант, который сочетает простоту Markdown с мощью React-компонентов. Next.js поддерживает формат MDX (Markdown + JSX), позволяя встраивать React-компоненты прямо в Markdown-файлы.

**Структура проекта:**

```text
app/
  [lang]/
    docs/
      [slug]/
        page.tsx        -- обёртка с layout, sidebar, поиском
content/
  en/
    docs/
      welcome.mdx      -- Markdown с встроенными React-компонентами
      install.mdx
      api-key-setup.mdx
  cn/
    docs/
      welcome.mdx      -- локализованная версия
      install.mdx
components/
  docs/
    Callout.tsx        -- кастомные компоненты для admonition
    CodeBlock.tsx      -- подсветка кода с copy-кнопкой
    StatusBadge.tsx    -- динамический виджет статуса
```

**Пример MDX-страницы:**

```mdx
---
title: Install ZCode
description: Download and install the new ZCode desktop app.
---

import { DownloadButton } from '@/components/docs/DownloadButton';
import { PlatformGrid } from '@/components/docs/PlatformGrid';

## Download

<DownloadButton version="latest" />

<PlatformGrid />

## Quick Installation Guide

### Windows

Download the installer, double-click it, and follow the setup wizard.

### macOS

Open the DMG file, drag `ZCode.app` into `Applications`.
```

Этот подход даёт лучшее из двух миров: авторы пишут в Markdown, но могут вставлять React-компоненты там, где нужна интерактивность или динамические данные.

#### 13.4.9. Извлечённые уроки

1. **Product-Embedded Docs оправданы только для SaaS-продуктов** -- если документация -- это часть пользовательского пути (onboarding, справка внутри интерфейса), инвестиция в встроенную документацию окупается. Для библиотек и open source-проектов это избыточно.

2. **MDX -- разумный компромисс** -- не обязательно писать каждую страницу как React-компонент. MDX позволяет сохранить скорость Markdown-редактирования, добавляя интерактивность точечно.

3. **Изоляция контента от кода** -- даже в Product-Embedded подходе стоит выносить тексты контента в отдельные файлы (MDX/JSON), а не встраивать их в компоненты. Это упростит локализацию и правки.

4. **Кэширование и производительность** -- встроенная документация рендерится сервером при каждом запросе (SSR) или генерируется статически (SSG). Для больших проектов предпочтительнее SSG с Incremental Static Regeneration (ISR), чтобы не нагружать сервер.

5. **Не забирайте у писателей возможность вносить правки** -- даже если документация встроена в продукт, обеспечьте интерфейс для нетехнических авторов: CMS-слой, headless CMS (Sanity, Contentful) или хотя бы MDX-файлы с простым форматом.
